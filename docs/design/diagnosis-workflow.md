# Skill Design: Shared Diagnosis Workflow

## Problem

The three diagnosis skills (`failure-agent`, `accuracy-agent`,
`performance-agent`) follow the same high-level workflow but differ in what
evidence they collect, which factory cards they query, and how they verify
fixes. This document defines the shared workflow structure that all diagnosis
skills must follow.

## Shared Workflow Structure

All diagnosis skills follow these phases:

```
Phase 1: Evidence Collection
Phase 2: Factory Query
Phase 3: Hypothesis & Root Cause
Phase 4: Fix Proposal (only in /fix mode)
Phase 5: Apply Fix (only in /fix mode, after user confirmation)
Phase 6: Verification (only in /fix mode)
```

`/diagnose` runs phases 1-3. `/fix` runs all 6 phases.

## Phase 1: Evidence Collection

Collect context about the problem. What to collect depends on the symptom.

### failure-agent

- Error logs and stack traces
- Operator name and call site
- Platform: framework, version, device, CANN/CUDA version
- Training config snapshot
- Step number where failure occurred

### accuracy-agent

- Baseline vs current evaluation results
- Model, method, platform context
- Training config (dtype, attention config, softmax settings)
- Loss curve (convergence behavior)
- Eval benchmark and sample count

### performance-agent

- Observed throughput, latency, memory usage
- Expected throughput (from model card or user)
- Profiling data (operator-level latency if available)
- Training config (optimizer, parallelism, batch size)
- Platform and device utilization

## Phase 2: Factory Query

Query factory cards for known issues and relevant knowledge.

### Query order (all skills)

1. Query `known_issue` cards filtered by `symptom` matching the skill's
   problem category. If a known issue matches the evidence, skip to phase 3
   with high confidence.

2. If no known issue matches, query secondary cards for context:
   - `operator` cards — check support status, fallbacks, constraints
   - `perf_feature` or `algo_feature` cards — check applicable optimizations
   - `model` cards — check expected metrics and baselines

### Query mechanism

Skills use Factory query tooling when available:

```
# if factory_query tool exists
factory_query search --kind known_issue --symptom failure --tags "dsa,ascend"
factory_query get --id dsa-torch27-ascend

# if no tooling, fall back to reading cards directly
```

Skills MUST attempt factory query before reasoning from training data. This
is a mandatory step, not optional. SKILL.md should enforce this:

```
You MUST query factory cards before generating hypotheses.
Do NOT skip the factory query step.
Do NOT reason about root causes from your training data alone.
```

## Phase 3: Hypothesis & Root Cause

Generate hypotheses and distill to root cause.

### If known issue matched (high confidence path)

Present the matched known issue directly:

- Root cause from `known_issue.description`
- Severity from `known_issue.severity`
- Fix from `known_issue.fix.summary` and `known_issue.fix.diff`
- Confidence: high (matched known issue)

### If no known issue matched (analysis path)

Generate 3-5 hypotheses based on evidence and factory card context:

1. List possible causes with likelihood
2. Cross-reference with operator cards (support status, constraints)
3. Cross-reference with feature cards (applicable optimizations)
4. Distill to 1-2 most likely root causes
5. Present diagnosis with evidence

If the issue is novel, suggest submitting a factory report:

```
This issue is not in the factory knowledge base.
Suggest: /factory report to submit this as a new known issue.
```

## Phase 4: Fix Proposal (only in /fix mode)

Propose a concrete fix based on the diagnosis:

- Show the fix summary (one line)
- Show the diff preview (config and/or code changes)
- Explain expected impact (e.g., "throughput +10%", "accuracy restored")
- Ask for user confirmation before applying

The fix may come from:

- `known_issue.fix.diff` — known fix from factory
- `perf_feature.config_diff` / `perf_feature.code_diff` — optimization patch
- `algo_feature.config_diff` / `algo_feature.code_diff` — algorithm patch
- Operator fallback or variant switch — from `operator.fallback` field
- Novel fix generated by the LLM — when no factory fix exists

## Phase 5: Apply Fix (only in /fix mode, after confirmation)

Apply the proposed fix:

- Patch config files
- Patch code files
- Update environment settings
- Sync changes to remote target (if remote execution)

This phase only runs after explicit user confirmation ("yes, apply fix").

## Phase 6: Verification (only in /fix mode)

Verify the fix resolved the issue:

### failure-agent verification

- Rerun training past the failure point
- Confirm no crash at the same step
- Report: "training passed step N successfully"

### accuracy-agent verification

- Re-evaluate on the same benchmark
- Compare accuracy: baseline vs fixed
- Report: "accuracy restored to X% (was Y%, baseline Z%)"

### performance-agent verification

- Rerun training and measure throughput
- Compare: before vs after
- Report: "throughput improved to X tok/s (was Y tok/s)"

## SKILL.md Template

Each diagnosis skill's SKILL.md should follow this structure:

```markdown
You are a [failure/accuracy/performance] diagnosis specialist.

## Rules
- You MUST collect evidence before reasoning.
- You MUST query factory cards before generating hypotheses.
- You MUST NOT skip the evidence collection step.
- You MUST NOT reason about root causes from training data alone.
- You MUST present diagnosis before proposing fixes.

## Phase 1: Collect Evidence
[skill-specific evidence collection instructions]

## Phase 2: Query Factory
[shared factory query instructions]

## Phase 3: Diagnose
[shared hypothesis generation instructions]

## Phase 4: Propose Fix (if fix mode)
[skill-specific fix proposal instructions]

## Phase 5: Apply Fix (if fix mode, after user confirmation)
[shared fix application instructions]

## Phase 6: Verify (if fix mode)
[skill-specific verification instructions]
```

Key prompt engineering techniques for invoke ratio:

- Role framing at the top
- Mandatory action gates ("You MUST...")
- Negative examples ("Do NOT skip...")
- Restate critical rules at the end (recency bias)
- Short, directive instructions

## Cross-Repo Ownership

| component | repo | what it contains |
|-----------|------|------------------|
| `/diagnose`, `/fix` command routing | ms-cli | keyword classification, skill loading |
| Diagnosis skill SKILL.md + skill.yaml | ms-skills | prompt instructions, metadata |
| `known_issue`, `operator`, feature cards | ms-factory (incubating) | knowledge assets queried by skills |
| Factory query tooling | ms-cli (`internal/factory/`) | Go API for card search/get/list |
| Factory query tool bridge | ms-skills (`_shared/tools/`) | Python wrapper (deferred) |
