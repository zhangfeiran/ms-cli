
PR #2 Review: version 2: Add basic agent core code

  Author: zhanghuiyao | +18,151 / -131 | 11 commits | 73 files

  This is a large PR that adds the core agent infrastructure: LLM provider, ReAct
  loop, tools, permission system, session/memory/context management, and TUI
  improvements. Here's the review:

  ---
  CRITICAL

  1. Shell Command Injection — tools/shell/runner.go
  Commands from the LLM are passed directly to sh -c without sanitization.
  SanitizePath() is defined but never called anywhere. The blocklist approach (; & |
  \``) is also fundamentally insufficient — it misses $()`, newlines, heredocs, etc.

  2. API Key Persisted in Plaintext — configs/state.go
  The API key is saved to .mscli/state.yaml in plaintext via SetKey() → Save(). Could
   be accidentally committed, leaked via backups, or read by any same-user process.
  Consider OS keychain or at minimum warn the user.

  ---
  HIGH

  3. Dangerous Command Detection Bypass — agent/loop/dangerous.go
  - Regexes are recompiled on every call instead of once at init
  - Easily bypassed via variable indirection (x="rm"; $x -rf /), base64, or paths
  (/bin/rm)
  - Fork bomb pattern :(){ :|:& };: contains unescaped regex metacharacters — will
  silently fail to compile, never matching

  4. Path Traversal Prefix Check Bug — tools/fs/*.go
  if !strings.HasPrefix(fullPath, t.workDir) {
  If workDir is /home/user/project, a path resolving to /home/user/projectevil passes
   the check. Fix: append filepath.Separator to the prefix. Also, symlinks are not
  resolved before checking.

  5. Shell Runner Pipe Race — tools/shell/runner.go
  The timeout branch recreates the cmd after StdoutPipe/StderrPipe were already
  attached to the original cmd. The new command has no pipes; goroutines read from
  orphaned pipes.

  6. Permissions Default to Allow Without UI — agent/loop/permission.go
  // No UI, default to allow
  return true, nil
  In headless/non-interactive environments, all PermissionAsk operations are silently
   auto-approved. Should default to deny.

  7. AddMessage Errors Silently Ignored — agent/loop/engine.go
  ctxManager.AddMessage() returns error but is called 4+ times without checking the
  return value. Failed compaction silently drops messages.

  ---
  MEDIUM

  8. Engine Not Safe for Concurrent Use — Shared ctxManager across all Run() calls
  means messages from different tasks interleave.

  9. Permission Store Written World-Readable — permission_store.go uses 0644; should
  be 0600 like the state file.

  10. unsafe Used Unnecessarily — agent/memory/sqlite_store.go uses unsafe.Pointer
  for float32↔byte conversion. Use math.Float32bits/math.Float32frombits instead.

  11. Dual Incompatible Tool Interfaces — agent/loop/executor.go and tools/types.go
  define different Tool interfaces requiring adapter structs. Should unify.

  12. Silent JSON Unmarshal Failures — agent/memory/sqlite_store.go ignores
  json.Unmarshal errors for metadata/tags. Corrupted data returns empty fields
  silently.

  13. Config Validate() Doesn't Check API Key — Validation passes but app fails later
   in initProvider().

  ---
  LOW

  - Bubble sort in session/file_store.go ListFiltered — use sort.Slice
  - panic in constructor (session/manager.go) — should return error
  - No output size limiting on Read/Grep/Glob tools — could overwhelm context window
  - No ctx.Done() checks in Grep/Glob during recursive walks
  - Mixed Chinese/English comments — should standardize
  - 5+ large markdown planning docs (ARCHITECTURE_ANALYSIS.md, DEVELOPMENT_PLAN.md,
  etc.) committed — these look like working notes, not user-facing docs. Consider
  removing or moving to a wiki.
  - Package-level mutable globals (executorRun, GlobalChecker) without
  synchronization

  ---
  Summary

  The architecture is reasonable — clean separation of agent loop, tools,
  context/memory, and UI. However, the PR has several security issues that should be
  addressed before merge, particularly around shell command injection, path
  traversal, and the permission system defaults. The concurrency bugs in the shell
  runner and the silently-ignored errors in the engine are also high priority.
